> 외주 작업 인계용 문서 (v0.1 — 초안)
> 작성일: 2026-06-13
> 범위: 와이어프레임 기반 **프론트엔드 UI 페이지 구현** (퍼블리싱)

## 0. 이 문서의 목적

이 문서는 전달받은 와이어프레임(Figma)을 기반으로 프론트엔드 페이지를
구현하면서 **"왜 이렇게 만들었는지"** 를 발주처가 이해할 수 있도록 정리한 기획서입니다.
코드를 받은 쪽에서 구조·기술 선택의 의도를 빠르게 파악하고,
필요하면 자신들의 코드베이스로 **떼어다 이식**할 수 있게 하는 것이 목표입니다.

## 2. 기술 스택과 선택 이유

| 영역 | 선택 | 이유 |
|------|------|------|
| 빌드 | **Vite** | 설정이 가볍고 빠름. 단순 페이지 구현에 SSR/풀스택 프레임워크(Next.js)는 과함. 발주처가 어떤 번들러를 쓰든 컴포넌트 단위 이식이 쉬움 |
| 언어 | **TypeScript** | props/타입이 명시되어 인계받는 쪽이 컴포넌트 사용법을 코드만 보고 파악 가능. 협업/병합 시 인터페이스가 곧 문서 역할 |
| UI 라이브러리 | **React** | 사실상 표준. 컴포넌트 단위 재사용이 Atomic Design과 잘 맞음 |
| 스타일 | **CSS Modules** | 컴포넌트별로 스타일이 **격리**되어 전역 클래스 충돌이 없음. 순수 CSS라 발주처가 어떤 스타일 스택(Tailwind/SCSS/styled-components)을 쓰든 떼어다 옮기기 가장 무난함. 런타임 의존성도 없음 |

### 왜 "병합 우선"이 아니라 "이식 우선"인가
현재 발주처의 기존 코드베이스/컨벤션 정보가 없습니다(9번 참고).
따라서 특정 레포에 곧바로 `git merge` 하는 것을 전제하지 않고,
**컴포넌트 단위로 깔끔하게 분리되어 "떼어다 쓰기 좋은" 결과물**을 목표로 합니다.
이를 위해 ① 스타일 격리, ② UI/프레젠테이션 위주(데이터 로직 분리),
③ 색·간격·폰트의 디자인 토큰(변수)화를 적용합니다.

## 3. 구조: Atomic Design

컴포넌트를 작은 단위부터 조립하는 방법론입니다. (레고 블록과 같은 사고방식)

| 단계 | 의미 | 예시 |
|------|------|------|
| **Atoms** | 더 못 쪼개는 최소 단위 | Button, Input, Label, Icon |
| **Molecules** | atom을 묶은 단위 | LabeledInput(라벨+입력칸), SearchBar |
| **Organisms** | molecule를 묶은 큰 덩어리 | PaymentMethodCard, Header |
| **Templates** | 데이터 없는 페이지 뼈대(레이아웃) | CheckoutLayout |
| **Pages** | 실제 데이터를 넣은 완성 화면 | CheckoutPage |

**효과:** 버튼 하나를 잘 만들면 모든 페이지에서 재사용 → 반복 코드 제거,
디자인 토큰만 바꿔도 전체 일괄 변경 → 유지보수/이식 용이.

### 제안 폴더 구조
```
src/
├─ components/
│  ├─ atoms/        # Button, Input, Icon ...
│  ├─ molecules/    # LabeledInput, ...
│  └─ organisms/    # Header, PaymentMethodCard, ...
├─ templates/       # 페이지 레이아웃 뼈대
├─ pages/           # 실제 화면 (라우트 단위)
├─ styles/
│  ├─ tokens.css    # 색/폰트/간격 등 디자인 토큰 (CSS 변수)
│  └─ global.css    # 리셋·기본 스타일
└─ types/           # 공용 타입
```

## 4. 디자인 토큰 방침

Figma에서 추출한 색상·폰트·간격·radius 등을 `styles/tokens.css`에
CSS 변수로 정의하고, 컴포넌트에서는 하드코딩 대신 변수를 참조합니다.
→ 발주처가 자신들의 테마 값으로 교체하기 쉽습니다.

```css
/* 예시 (실제 값은 Figma 분석 후 확정) */
:root {
  --color-primary: #...;
  --space-md: 16px;
  --radius-sm: 8px;
}
```

## 5. 반응형 방침

- 발주처가 반응형 범위를 명시하지 않아, **휴리스틱하게 적용**합니다.
  (결제 페이지 특성상 모바일 사용 비중이 높아 모바일 대응을 기본 고려)
- 반응형 때문에 들어간 코드(미디어쿼리, flex 정렬 전환 등)에는
  **"왜 반응형 처리인지"를 주석으로 명시**하여 인계받는 쪽이 의도를 알 수 있게 합니다.

```css
/* 반응형: 모바일(<768px)에서 가로 정렬 → 세로 정렬로 전환 */
@media (max-width: 768px) { ... }
```

## 6. 코드 품질 / 인계 방침

- props/타입을 명확히 → 컴포넌트 사용법이 코드로 드러나게
- 컴포넌트는 단일 책임, 파일명·디렉터리 일관성 유지
- 주석은 "무엇"이 아니라 **"왜"** 위주로 (특히 반응형·예외 처리)
- 비즈니스 로직과 UI를 분리해 발주처 데이터 흐름에 덜 엮이게

### 8.3 공통 컴포넌트 후보 (반복 구조)
- **레이아웃**: `역할별 사이드바 / leader`(300px 고정), `콘텐츠` 영역, `역할 포털 네온 글로우`(배경 장식)
- **공통 요소**: 상태 배지(승인요청/검토중/자료요청/보류/승인/거절/대기/활성/운영정지/블랙리스트), 데이터 테이블, 검색/필터 바, 버튼(취소/요청/저장), 입력 필드, 토스트, 모달, KPI 카드
- **구현된 공통 컴포넌트(추가)**: `DetailDrawer`(organism, 우측 슬라이드 상세 드로어 — 목록 "상세" 진입 공용), `DetailSection`(molecule, 드로어 섹션 박스: 제목+라벨/값 줄)
